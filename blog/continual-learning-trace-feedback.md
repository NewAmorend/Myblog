> Trace 并非全部。Trace 只记录了「发生了什么」，必须结合 Feedback 才能成为后续 RL 的燃料。

## Trace ≠ 足够

Trace 能告诉你 Agent 做了什么，但不能告诉你「做得好不好」。
没有 Feedback 的 Trace，就像没有标签的训练数据——有结构，但没有信号。

## 一个工程上的错配

### 软件工程师眼中的「好」

- 能连通
- 低延迟
- MTTF（平均无故障时间）低

这是传统的软件质量指标。

### 用户眼中的「好」

- 这个回答对我有没有帮助？
- 解决了我的问题没有？
- 我需不需要反复追问？

**核心矛盾：** 软件工程视角的「好」不能真正反映用户觉得你这个 Agent 有没有用。一个延迟很低但答非所问的 Agent，用户照样骂娘。

## Feedback 的归属难题

### 问题

当 Agent 表现不好时，到底是哪一层出了问题？

**案例：DeerFlow 的无限循环工具调用 bug**

可能原因：
1. **Harness 层问题** — 循环检测中间件设计缺陷
2. **Model 层问题** — 早期模型（如 dsv3.2）太笨，不会正确使用 harness
3. **两者皆有** — harness 设计不够防呆，模型能力又不够

### 现有解法

| 方法 | 优点 | 缺点 |
|------|------|------|
| **人工 A/B 实验** | 最可靠 | 成本高、慢 |
| **LLM-as-Judger** | 可规模化 | 框架设计本身需要大量人工经验 |
| **自动规则（REGEXP 等）** | 即时、零成本 | 覆盖面窄，只能捕捉极端情况 |

### 目前的判断

> 这部分目前还是得靠人工。收集到足够多的数据之后，再尝试 LLM 介入。

LLM-as-Judger 的框架如何设计，本身可能又需要大量人工判断的经验来校准。

## 一个有意思的发现：REGEXP 直接检测

除了用户反馈和 LLM-as-Judger，还有一个低成本高信号的方法：

**直接用正则匹配骂人词汇。**

```
用户骂娘了 → 反馈肯定差
```

这看起来很粗暴，但其实是：
- **高精度的负样本信号** — 只要用户开始骂人，几乎100%说明体验极差
- **零延迟** — 不需要等 LLM 评估，实时检测
- **实现成本极低** — 几行正则搞定

可以维护一个「情绪黑名单词汇表」，检测到就自动标记为负反馈，作为 RL 训练的负样本。

### 更广义的情绪信号

| 信号 | 含义 | 检测方式 |
|------|------|----------|
| 骂人词汇 | 极端不满 | REGEXP |
| 重复提问同一问题 | Agent 没理解用户意图 | 语义相似度 |
| 用户中途放弃对话 | 失去耐心 | 对话长度 + 最后一条是用户消息 |
| 用户手动修改 Agent 输出 | Agent 结果不可用 | diff 比较 |
| 用户给出好评/感谢 | 正面反馈 | REGEXP + 情绪分析 |

---

## 小结

```
Trace（发生了什么）+ Feedback（好不好）= RL 的燃料
```

但 Feedback 的归属（到底是哪一层的问题）目前仍是开放难题，短期内依赖人工，长期可能通过 LLM-as-Judger + 大量标注数据逐步自动化。
